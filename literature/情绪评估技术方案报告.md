# OpenClaw记忆引擎情绪评估模块技术方案报告

## 背景
基于PAD情绪模型、Lazarus认知评价理论和Zajonc优先机制，为OpenClaw记忆引擎设计情绪评估模块。需要从文本对话内容提取情绪维度（愉悦度、唤醒度、优势度），并集成到现有记忆引擎架构中。

## 1. 轻量级情绪分析库评估

### TextBlob
- **功能**：情感极性分析（正/负），主观性分析
- **优点**：简单易用，安装方便，英文处理成熟
- **缺点**：不支持中文，需要翻译预处理；仅提供极性，不提供唤醒度和优势度
- **适用场景**：英文文本的快速情感分析，不适用于中文环境

### VADER (Valence Aware Dictionary and sEntiment Reasoner)
- **功能**：社交媒体情感分析，输出正/负/中性分数及复合分数
- **优点**：对社交媒体语言（缩写、表情符号）优化，无需训练
- **缺点**：仅支持英文，同样缺乏唤醒度和优势度维度
- **适用场景**：英文社交媒体内容分析

### NRC Emotion Lexicon
- **功能**：情感词典，关联单词与8种基本情绪（愤怒、期待、厌恶、恐惧、喜悦、悲伤、惊讶、信任）和两种情感（正/负）
- **优点**：多语言支持（包括中文），资源公开
- **缺点**：仅单词级别，需要文本分割和聚合；无连续维度输出
- **适用场景**：基于词典的情绪分类，可通过映射表转换为PAD评分

### 中文情绪分析库推荐
1. **SnowNLP**：中文情感分析库，基于朴素贝叶斯，可输出情感极性分数
2. **senta**（百度开源）：基于深度学习的情绪分析，支持多种情绪分类
3. **TencentCloud NLP** / **Aliyun NLP**：商业API，功能强大但需要网络和付费
4. **jieba** + **自定义情感词典**：自行构建情感词典，结合PAD映射

### 轻量级方案总结
| 库 | 中文支持 | PAD映射难度 | 计算开销 | 准确性 | 推荐度 |
|----|----------|-------------|----------|--------|--------|
| TextBlob | 否 | 高 | 低 | 低（中文） | 不推荐 |
| VADER | 否 | 高 | 低 | 低（中文） | 不推荐 |
| NRC Emotion Lexicon | 是 | 中 | 低 | 中 | 可考虑 |
| SnowNLP | 是 | 中 | 低 | 中 | 推荐 |
| senta | 是 | 中 | 中 | 中高 | 推荐 |

## 2. 基于Transformer的小型模型评估

### BERT微调
- **可行性**：需要标注PAD三维评分的中文数据集，目前公开数据集稀缺
- **数据需求**：至少数千条标注样本
- **性能预期**：准确度高，可端到端输出PAD连续值
- **计算开销**：推理速度较慢，需要GPU或高性能CPU
- **推荐模型**：BERT-wwm-ext, RoBERTa-wwm-ext

### DistilBERT
- **特点**：BERT的轻量级版本，参数减少40%，推理速度提升60%
- **效果对比**：准确度略低于BERT，但计算开销显著降低
- **适用性**：适合资源受限环境，仍需要微调数据

### 其他小型Transformer模型
- **RoBERTa-tiny**：参数更少，适合移动端
- **ALBERT-base**：参数共享，内存占用小
- **Chinese-BERT-wwm**：针对中文优化，有small版本

### 预训练情绪分析模型
- 可考虑使用已在中文情绪分析任务上微调的预训练模型，如`uer/roberta-base-finetuned-chinanews-sentiment`
- 但需要转换为PAD维度，可能需要额外回归层

## 3. 规则与统计结合的混合评估方法

### 混合架构设计
```
文本输入
    ↓
[统计层]：情绪分析模型 → 初始PAD评分
    ↓
[规则层]：Lazarus认知评价框架 → 情境调整
    ↓
[快速路径]：Zajonc优先机制 → 直觉情绪标记
    ↓
最终PAD评分
```

### 规则层实现
基于Lazarus的初级评价（目标相关性）和次级评价（应对资源）：
- **目标相关性**：匹配记忆内容与用户目标关键词，增加唤醒度
- **应对资源**：根据用户历史应对能力调整优势度
- **情境匹配**：结合当前对话上下文调整愉悦度

### 快速路径实现
- 关键词匹配高频情绪词（如“开心”、“愤怒”），直接赋予预设PAD评分
- 响应时间<100ms，适用于实时对话

### 混合评估优势
- 结合数据驱动与理论驱动，提高解释性
- 降低对标注数据的依赖
- 适应动态情境变化

## 4. 技术实现考虑

### 实现复杂度（低→高）
1. **关键词匹配**：最简单，基于情感词典
2. **轻量级库**：中等，调用现有API
3. **混合评估**：较高，需要集成规则引擎
4. **Transformer微调**：最高，需要数据收集、训练、部署

### 计算开销（低→高）
1. **关键词匹配**：CPU低负载，内存占用小
2. **轻量级库**：CPU中等，依赖库大小
3. **混合评估**：CPU中等，规则推理开销
4. **Transformer微调**：GPU/高CPU，内存占用大（>500MB）

### 准确性（低→高）
1. **关键词匹配**：低，忽略上下文
2. **轻量级库**：中，对简单文本有效
3. **混合评估**：中高，结合情境
4. **Transformer微调**：高，端到端学习

### 可维护性（易→难）
1. **关键词匹配**：易维护，规则透明
2. **轻量级库**：易，依赖库更新
3. **混合评估**：中等，规则与模型分离
4. **Transformer微调**：难，需要持续训练数据

## 5. 综合因素评估与选型建议

### 推荐方案：混合评估（轻量级库 + 规则引擎）
**理由**：
1. **平衡性能与开销**：使用轻量级中文情绪分析库（SnowNLP或senta）提供基础情感评分，结合规则引擎映射到PAD三维
2. **适应OpenClaw环境**：无需GPU，可在普通笔记本上运行
3. **可扩展性**：未来可替换为Transformer模型提升准确度
4. **理论符合**：集成Lazarus和Zajonc机制，符合科学理论

### 降级替代方案
如果计算资源极度受限：
- **纯关键词匹配**：使用NRC Emotion Lexicon中文扩展词典，构建词到PAD的映射表
- **简化规则**：仅基于唤醒度调整记忆权重，忽略愉悦度和优势度

### 简化代码示例
```python
import numpy as np
from snownlp import SnowNLP

class EmotionAnalyzer:
    def __init__(self):
        # 情感词典映射：词 -> (pleasure, arousal, dominance)
        self.emotion_lexicon = {
            "开心": (0.8, 0.6, 0.4),
            "愤怒": (-0.7, 0.9, -0.3),
            # ... 更多词条
        }
    
    def analyze_text(self, text):
        # 统计层：SnowNLP情感分析
        s = SnowNLP(text)
        sentiment = s.sentiments  # 0~1，正面概率
        
        # 映射到PAD基础评分
        pleasure = sentiment * 2 - 1  # [-1, 1]
        arousal = 0.5  # 默认中等唤醒
        dominance = 0.0  # 默认中性
        
        # 规则层：Lazarus情境调整
        pleasure, arousal, dominance = self._contextual_adjustment(
            text, pleasure, arousal, dominance
        )
        
        # 快速路径：关键词匹配
        for word, pad in self.emotion_lexicon.items():
            if word in text:
                pleasure = 0.7 * pleasure + 0.3 * pad[0]
                arousal = 0.7 * arousal + 0.3 * pad[1]
                dominance = 0.7 * dominance + 0.3 * pad[2]
                break
        
        return np.clip([pleasure, arousal, dominance], -1, 1)
    
    def _contextual_adjustment(self, text, p, a, d):
        # 基于Lazarus规则的调整
        # 示例：检测目标关键词增加唤醒度
        goal_keywords = ["目标", "任务", "必须"]
        if any(k in text for k in goal_keywords):
            a = min(a + 0.2, 1.0)
        
        # 检测应对资源关键词调整优势度
        resource_keywords = ["能够", "可以", "成功"]
        if any(k in text for k in resource_keywords):
            d = min(d + 0.3, 1.0)
        
        return np.array([pleasure, arousal, dominance])
    
    def _contextual_adjustment(self, text, p, a, d):
        # 更复杂的规则实现
        # 可结合记忆上下文、用户状态等
        return p, a, d
```

## 6. 集成到现有记忆引擎

### 情绪数据格式
在记忆项数据结构中添加情绪相关字段：
```json
{
  "id": "20260328_001",
  "content": "今天学习了量子计算，感觉很有成就感",
  "vec": [...],
  "importance": 0.8,
  "type": "knowledge",
  "timestamp": "2026-03-28T10:45:00",
  "emotion": {
    "pad": [0.7, 0.6, 0.4],  // pleasure, arousal, dominance
    "source": "text_analysis",
    "confidence": 0.85,
    "last_updated": "2026-03-28T10:45:00"
  }
}
```

### 集成点
1. **记忆摄入时**：`ingest_log_file`方法中对每个chunk调用情绪分析
2. **记忆检索时**：`retrieve`方法中考虑情绪权重，高唤醒度记忆得分加成
3. **睡眠周期**：情绪评分作为记忆重要性评估的输入之一

### 性能优化技巧
- **缓存情绪分析结果**：相同文本哈希缓存PAD评分，避免重复计算
- **异步分析**：非实时场景可将情绪分析任务放入队列
- **批量处理**：多个记忆项一起分析，减少模型加载开销
- **降级策略**：当情绪分析服务不可用时，使用默认中性评分

## 8. 环境限制与应对措施

### OpenClaw环境限制
- **计算资源有限**：可能无法运行大型Transformer模型
- **网络连接不稳定**：无法依赖云端API
- **存储空间有限**：模型文件不宜过大

### 应对措施
1. **SnowNLP离线使用**：提前下载训练数据（`sentiment.marshal`和`sentiment.marshal.3`），确保无网络环境下可用
2. **模型轻量化**：可选择DistilBERT或ALBERT-tiny作为未来升级目标，模型大小<100MB
3. **缓存机制**：避免重复分析，减少计算开销
4. **降级策略**：当情绪分析失败时，使用基于关键词的默认评分
